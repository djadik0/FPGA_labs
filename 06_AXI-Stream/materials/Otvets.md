## Ответы на контрольные вопросы

### 1. Зачем нужен сигнал `ready` в интерфейсе `valid-ready`?

Сигнал `ready` показывает, что принимающая сторона готова принять данные в текущем такте.  
Он нужен для управления потоком данных без потерь: источник может держать данные и `valid = 1`, пока приёмник не подтвердит готовность через `ready = 1`.

---

### 2. Что такое рукопожатие (`handshake`)?

Рукопожатие — это момент успешной передачи данных, когда одновременно:

- `valid = 1`
- `ready = 1`

Именно в этот такт данные считаются переданными от источника к приёмнику.

---

### 3. Что такое обратное давление (`backpressure`)?

`Backpressure` — это механизм, при котором приёмник может временно остановить поток данных, сняв сигнал `ready`.  
Так источник понимает, что передавать новые данные пока нельзя.

Это позволяет:
- не терять данные;
- согласовывать скорость работы разных блоков;
- останавливать конвейер при переполнении буферов.

---

### 4. Какая проблема возникает с сигналом `ready` при соединении большого количества модулей в конвейер?

В длинном конвейере сигнал `ready` часто распространяется **в обратную сторону** через много модулей сразу.  
Из-за этого возникает длинный **комбинационный критический путь** по `ready`.

Проблемы:
- ухудшается тайминг;
- падает максимальная частота;
- усложняется трассировка и синтез;
- растёт зависимость между удалёнными стадиями конвейера.

---

### 5. Какие способы развязки критического пути по `backpressure` вы знаете? Опишите их.

#### 1. FIFO-буфер

Между стадиями ставится FIFO.  
Он накапливает данные и разрывает прямую зависимость между upstream и downstream блоками.

Плюсы:
- простая реализация;
- хорошо разрывает путь по `ready`;
- позволяет буферизовать поток.

#### 2. Регистровый slice / pipeline register

Между модулями добавляется промежуточный регистр для данных и сигналов управления.  
Так сигнал `ready` уже не проходит через весь тракт за один такт.

Плюсы:
- уменьшает длину критического пути;
- делает конвейер более регулярным.

#### 3. Credit-based flow control

Вместо прямого распространения `ready` используется счётчик доступных мест — кредиты.  
Источник заранее знает, сколько транзакций можно отправить без немедленного ответа от приёмника.

Плюсы:
- хорошо подходит для длинных трактов;
- убирает длинную обратную цепь `ready`.

#### 4. Skid buffer / elastic buffer

Небольшой буфер на 1–2 слова, который позволяет временно удержать данные при резком снятии `ready`.

Плюсы:
- дешёвый по ресурсам;
- помогает сохранить данные без потери;
- разрывает часть критического пути.

---

### 6. Что такое кредитный механизм? Для чего он нужен? Как он работает?

Кредитный механизм — это способ управления потоком данных, при котором источник не ждёт постоянный `ready`, а ориентируется на число доступных мест у приёмника.

#### Для чего нужен

- для развязки длинного пути по `ready`;
- для повышения частоты;
- для устойчивой передачи данных в глубоких конвейерах и межмодульных соединениях.

#### Как работает

1. У источника есть счётчик кредитов.
2. Каждый кредит означает, что приёмник может принять ещё одну транзакцию.
3. При отправке данных один кредит тратится.
4. Когда приёмник освобождает место, он возвращает кредит обратно.
5. Если кредиты закончились, источник должен остановить передачу.

То есть передача разрешена, пока есть свободные кредиты.

### 7. Что такое интерфейс `AXI Stream`? Чем он отличается от интерфейса `valid-ready`?

`AXI Stream` — это потоковый интерфейс передачи данных, построенный на той же идее, что и `valid-ready`, но расширенный дополнительными служебными сигналами.

#### Отличие от обычного `valid-ready`

Базовый `valid-ready` обычно содержит только:
- данные;
- `valid`;
- `ready`.

`AXI Stream` добавляет стандартный набор сигналов для:
- границ пакетов;
- маршрутизации;
- маскирования байт;
- передачи служебной информации.

То есть `AXI Stream` — это более формализованный и расширенный потоковый интерфейс.

---

### 8. Какие сигналы есть в интерфейсе `AXI Stream`?

Основные сигналы:

- `TVALID` — данные на шине валидны;
- `TREADY` — приёмник готов принять данные;
- `TDATA` — передаваемые данные.

Дополнительные сигналы:

- `TLAST` — признак последнего слова пакета;
- `TKEEP` — какие байты `TDATA` являются значимыми;
- `TSTRB` — строб по байтам;
- `TID` — идентификатор потока;
- `TDEST` — идентификатор назначения, используется для маршрутизации;
- `TUSER` — пользовательская служебная информация.

### 9. Объясните принцип разветвления потоков данных (`fork`)

`Fork` — это разветвление одного входного потока на несколько выходных направлений.

Есть два основных варианта:

#### 1. Копирование потока на несколько выходов

Одни и те же данные передаются сразу в несколько направлений.

#### 2. Маршрутизация в один из выходов

Для каждой транзакции выбирается только один выход, например по `TDEST`.

#### Особенность

Нужно правильно сформировать `valid` и `ready`, чтобы:
- не потерять данные;
- не отправить их не туда;
- не нарушить handshake.

Если это fork с копированием, то часто требуется дождаться готовности всех нужных получателей.

---

### 10. Объясните принцип слияния потоков данных (`join`). Какие сложности возникают при слиянии потоков данных?

`Join` — это объединение нескольких входных потоков в один выходной.

Есть два распространённых случая.

#### 1. Объединение независимых потоков

Из нескольких входов выбирается один активный, и его данные передаются на общий выход.

#### 2. Синхронное объединение

Нужно дождаться данных сразу с нескольких входов и затем сформировать общий результат.

#### Основные сложности

- одновременно могут быть активны несколько входов;
- нужно решать, какой вход обслуживать первым;
- нельзя потерять данные у невыбранных входов;
- нужно корректно формировать `ready` только для выбранного входа;
- возможны блокировки и падение пропускной способности;
- при синхронном join нужно ждать все нужные входы, что усложняет логику.

### 11. Что такое алгоритм арбитража? Какие алгоритмы арбитража вы знаете? Опишите их.

Алгоритм арбитража — это правило выбора одного запроса из нескольких одновременно активных.

Он нужен, когда несколько источников хотят использовать один общий ресурс:
- выходной канал;
- память;
- шину;
- вычислительный блок.

#### Основные алгоритмы арбитража

##### 1. Статический приоритет

Каждому входу заранее задан приоритет.  
Если активны несколько запросов, выбирается вход с наивысшим приоритетом.

Плюсы:
- очень простой;
- быстрый.

Минусы:
- возможна starvation — входы с низким приоритетом могут долго не обслуживаться.

##### 2. Round-robin

После каждого обслуживания приоритет циклически переходит к следующему входу.

Плюсы:
- более справедливый;
- снижает вероятность голодания.

Минусы:
- логика сложнее, чем у статического приоритета.

##### 3. Masked round-robin

Похоже на round-robin, но используется маска или указатель текущей позиции для выбора следующего запроса.

Плюсы:
- удобен для аппаратной реализации;
- обеспечивает справедливость.

##### 4. Случайный выбор

При нескольких запросах выбирается случайный вход.

Плюсы:
- простая идея;
- в некоторых задачах даёт равномерность в среднем.

Минусы:
- нет строгой гарантии справедливости.

##### 5. Взвешенный арбитраж

Разным входам назначаются разные веса или квоты обслуживания.

Плюсы:
- можно гибко распределять пропускную способность.

Минусы:
- логика сложнее.

### 12. Что такое коммутатор? Из каких частей состоит коммутатор и как он построен?

Коммутатор — это устройство или модуль, который направляет входные потоки данных на нужные выходы по заданному правилу маршрутизации.

В потоковых интерфейсах коммутатор обычно нужен, чтобы:
- принять несколько входных потоков;
- выбрать нужный источник;
- направить данные в нужный выходной канал.

#### Из каких частей состоит коммутатор

Обычно коммутатор включает:

##### 1. Блоки объединения (`join`)
Они собирают несколько входных потоков в один общий поток.

##### 2. Арбитр
Он решает, какой из входов сейчас получит доступ к общему тракту.

##### 3. Блок маршрутизации (`fork` / demux)
Он направляет выбранный поток на нужный выход.

##### 4. Логику управления `valid/ready`
Она обеспечивает корректное handshake-взаимодействие между всеми частями.

#### Как он построен

Типичная схема такая:

1. Несколько входов подаются на `join`.
2. Арбитр выбирает один активный вход.
3. Данные выбранного входа передаются дальше.
4. По полю маршрутизации, например `TDEST`, поток направляется на нужный выход через `fork`.
5. Сигналы `ready` возвращаются назад только по активному пути.

#### Идея работы

Коммутатор сочетает:
- **арбитраж** — кто сейчас передаёт;
- **маршрутизацию** — куда передавать;
- **управление потоком** — когда передавать.
