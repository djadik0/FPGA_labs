# Лабораторная работа 6. AXI-Stream, Valid-Ready, Credit Flow Control

## Цель работы

Изучить принципы потоковой передачи данных с использованием интерфейсов `valid-ready` и `AXI-Stream`, а также познакомиться со способами устранения backpressure и организации арбитража между несколькими потоками.

В рамках лабораторной работы необходимо:
- реализовать вычислительный блок с интерфейсом `valid-ready`;
- реализовать конвейер с интерфейсом `AXI-Stream` и кредитным механизмом;
- реализовать коммутатор потоков данных с арбитражом;
- проанализировать структуру модулей и особенности работы сигналов `valid`, `ready`, `TDEST` и `TUSER`.

## Краткая теория

### Интерфейс `valid-ready`

Интерфейс `valid-ready` используется для потоковой передачи данных между двумя блоками цифровой схемы.

Передача данных выполняется только в тот такт, когда одновременно активны:
- `valid = 1` — источник выставил корректные данные;
- `ready = 1` — приёмник готов принять данные.

Такой момент называется **рукопожатием** (`handshake`).

Наличие сигнала `ready` позволяет реализовать механизм **backpressure**, при котором принимающая сторона может временно остановить продвижение данных по тракту.

### FIFO и credit flow control

Для разрыва критического пути по линии `ready` и буферизации данных могут использоваться FIFO.

Кредитный механизм основан на том, что число допустимых транзакций в конвейере ограничено, а освобождение места на приёмной стороне приводит к возврату кредита источнику.  
Такой подход позволяет организовать устойчивую потоковую передачу без потери данных.

### AXI-Stream

AXI-Stream — это потоковый интерфейс, основанный на той же идее `valid-ready`, но расширенный набором дополнительных сигналов.

Основные сигналы интерфейса:
- `TVALID`
- `TREADY`
- `TDATA`

Дополнительно могут использоваться:
- `TLAST`
- `TKEEP`
- `TSTRB`
- `TID`
- `TDEST`
- `TUSER`

В данной лабораторной работе особое внимание уделяется использованию `TDEST` для маршрутизации транзакций и `TUSER` для передачи дополнительной служебной информации.

## Выполненные индивидуальные задания

### Задание 1

Был реализован блок с тремя входными потоками и одним выходным потоком с интерфейсом `valid-ready`.

Реализуемая функция:

```text
y = a² + b² + c²
```

### Особенности реализации

- используются три входных канала данных: `a`, `b`, `c`;
- вычисление выполняется только после появления валидных данных сразу на всех трёх входах;
- выходной результат передаётся через интерфейс `valid-ready`;
- ширина выходных данных в реализованном модуле составляет `32` бита.

В файле `task1.sv` реализован модуль `task1`, в котором:
- сигналы `ready_a_o`, `ready_b_o` и `ready_c_o` зависят от готовности выходного интерфейса;
- результат вычисления сохраняется во внутреннем регистре;
- признак валидности результата хранится отдельно.

## Задание 2

Был реализован конвейер с интерфейсом AXI-Stream, выполняющий цепочку вычислений по варианту:

```text
стадия 1: -2
стадия 2: + значение 3 транзакции назад
стадия 3: * 9
стадия 4: + 76
стадия 5: % 1024
```

### Особенности реализации

- входная ширина данных — `16` бит;
- между стадиями используются FIFO-обёртки;
- глубина FIFO соответствует `8` элементам;
- на второй стадии используется история значений на `3` транзакции назад;
- выходная ширина данных в реализованном модуле составляет `10` бит, что соответствует операции `%1024`.

В файле `task2.sv` реализован модуль `task2`, который построен как пятистадийный конвейер с промежуточными FIFO:
- `fifo1`
- `fifo2`
- `fifo3`
- `fifo4`

Также в модуле используются:
- `fifo_wrapper.sv`
- `simple_fifo.sv`

## Задание 3

Был реализован коммутатор AXI-Stream со следующими параметрами:
- `4` входных порта;
- `6` выходных портов;
- статический арбитраж;
- порт `0` имеет наивысший приоритет.

### Особенности реализации

- входной порт выбирается по статическому приоритету;
- направление транзакции на один из выходов задаётся с помощью сигнала `TDEST`;
- для указания источника выбранной транзакции используется поле `TUSER`;
- готовность выбранного выходного порта влияет на сигнал `ready` соответствующего входа.

В файле `task3.sv` реализован модуль `axis_join_static`, в котором:
- входные запросы собираются в вектор `req`;
- решение арбитража формируется в векторе `grant`;
- данные направляются на один из шести выходов в зависимости от значения `TDEST`;
- арбитраж является статическим и отдаёт приоритет младшему входному порту.
